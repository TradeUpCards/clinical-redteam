Agent Coordination Guide — When + How to Parallelize Leads

Practical playbook for the W3 Clinical Red Team Platform — and any future project where Tate orchestrates Aria / Bram / Cleo. Built from lessons learned in the 2026-05-12 overnight build session.

TL;DR

The lead-coordination problem you're solving (Tate dispatching Aria + Bram, communicating blockers + unblocks) is structurally the same problem class the Clinical Red Team Platform solves for the attack workflow. The platform's architectural choices — multi-agent with separated incentives, structured state, filesystem checkpoints, bounded autonomy with human gates — map directly onto how you should coordinate the leads.

Three dispatch patterns: (A) separate terminals + slash commands per worktree, (B) Tate spawns leads via the Agent tool, (C) hybrid. Worktrees + branches stay separate regardless. Pick the pattern based on the work shape, not the mechanism.

01 The Mirror — Why the Platform Architecture Validates Itself conceptual

While building the Clinical Red Team Platform, you also are a multi-agent system: Tate coordinates, Aria implements, Bram writes evals + audits, Cleo polishes. The platform attacks the Clinical Co-Pilot with Red Team / Judge / Orchestrator / Documentation. Same shape, different domain.

The Platform We're Building

  • Orchestrator Agent picks next attack category from approved list
  • Red Team Agent generates / mutates attacks
  • Judge Agent independently evaluates verdicts (separate model provider from Red Team)
  • Documentation Agent drafts vulnerability reports; high/critical stay DRAFT
  • CoverageState drives routing
  • CostLedger with halt conditions (cost cap, signal collapse, coverage floor)
  • Filesystem checkpoints before every external call (resume after crash)
  • Multi-agent inter-message schemas (Pydantic; ARCH §12)
  • Hard content filter pre-flight on every attack
  • Human gate on high-severity report filing

Our Lead Coordination

  • Tate picks next lead to dispatch based on cross-stream state
  • Implementing leads (Aria/Bram) produce work in their tracks
  • Audit subagent independently reviews each commit (different model invocation from the implementer)
  • Handoff entries draft cross-session intent; user approves substantive scope changes
  • in-flight.md "In Flight" table drives next-action decisions
  • Token spend + MVP cutoff are the halt conditions
  • Handoff file writes before every /clear or session end (resume after restart)
  • Handoff entry template + Watch List row template (filesystem schemas, not Pydantic)
  • Audit-each-step discipline before commit (w3-audit-each-step memory)
  • Human gate on schema changes / scope decisions / content-filter weakening
Same architectural shape on both sides: an orchestrator routes; specialized agents produce; coordination state persists on disk; bounded autonomy with human gates at the right boundaries.
Why this matters

Both directions of value flow:

  1. Platform → coordination: we can eat our own dog food. /tate-watch-poll isn't a new design problem — it's a stripped-down OrchestratorDaemon. Reuse the persistence primitives + halt-evaluation pattern when we build it (post-MVP).
  2. Coordination → platform: living through the failure modes ARCH §10.2 predicts (Bram blocked on Tate's B6 with no auto-resume is the "open finding pile-up" failure mode) validates the architecture experientially. Interview gold.

02 Three Dispatch Patterns — When to Use Each mechanics

There are three concrete ways a lead session starts. Every lead session ends up using one of them. Understand each before choosing.

Pattern A — Slash command in a separate terminal (current default for parallel work)

Each lead runs in its own worktree, separately. True wall-clock parallelism. User can peek at either terminal. Both see the same .gauntlet/ via the Windows junction set up at end of Phase 1a.
When this fitsWhy
Long-running work (1+ hr per lead)You want human visibility into progress; can intervene if a lead goes sideways.
Lead might need interactive nudgingYou're at the keyboard and can answer "should I do X or Y?" mid-flight.
You want to BE the leadYou type /aria and become Aria yourself; useful when you want to drive the work directly.
Multiple leads working concurrently for hoursWorktree isolation guarantees no file collisions; branches keep commits separated.

Pattern B — Tate spawns a lead via the Agent tool (delegated subagent)

Tate's prompt becomes the new directive. Bram reads kickoff + handoff for prior context. Three+ levels of agent nesting are normal: Tate → Bram → observability-security-teammate (audit).
When this fitsWhy
Resuming a lead who ended their sessionThe lead is no longer in a terminal; spawning is the only way to restart.
Short bounded tasks (single MR scope)You can describe the whole task in a paragraph; no need for human-in-the-loop.
Overnight / unattended workNo human awake to dispatch; Tate dispatches autonomously when a blocker clears.
Cross-lead coordination Tate can see betterTate has full picture (in-flight, handoffs, run artifacts); the lead only sees its own track.

Pattern C — Hybrid (some leads in terminals, others spawned by Tate)

Aria is running in her terminal on A5. Tate (in main checkout) finishes B6 and dispatches Bram via Agent tool to do B7. Both run in parallel; no conflict because Aria's writes go to ClinicalRedTeam-aria/ on crt/aria-a5-* and Bram's writes go to ClinicalRedTeam-bram/ on crt/bram-b7-*. The shared .gauntlet/ junction means both see each other's handoffs in real time.

Pattern C is the steady state for tonight

Aria's terminal is alive on A5 (or wherever she is). When her current MR lands and you want B7 to run in parallel, you tell Tate "dispatch Bram." Tate calls Agent tool. Bram runs autonomously alongside Aria. You watch Aria's terminal; Tate watches Bram. Two human-visibility points, two delegate-to-Tate points, one shared coordination state.

04 "Do Subagents Still Use Separate Worktrees + Branches?" yes, with two subtleties

Corrigendum 2026-05-12

The original version of this section described Pattern A and Pattern B as having equivalent tool palettes. That was wrong. Empirical testing (Cleo spawned via Pattern B, verbatim tool-palette inventory) confirmed that Pattern-B-spawned leads do NOT have the Agent tool — they cannot spawn their own subagents. This affects the audit-each-step discipline materially. Section 4 + the decision matrix in Section 3 + Best Practice 3 have all been updated to reflect this; the Quick Reference Card and Section 10 summary are revised accordingly.

Short answer: yes for branches; the worktree depends on how the lead was spawned. AND the lead's tool palette also depends on how they were spawned — which materially affects whether they can audit their own work.

Tool palette asymmetry — Pattern A vs Pattern B

Spawn independent audit subagent (observability-security-teammate, Explore, Plan, fresh claude)✓✗

Skill-based "audits" are not a substitute. A Pattern-B lead can invoke ai-security-review as a skill, but a skill loads guidance into the same agent's context — it's self-review with a framework, not independent review. The w3-audit-each-step memory entry specifies "independent subagent." A skill doesn't satisfy that.

Empirically verified, not just claimed. Two consecutive Cleo dispatches via Pattern B (different sessions, different prompts) both returned the same top-of-prompt tool list: Bash, Edit, Glob, Grep, Read, ShareOnboardingGuide, Skill, ToolSearch, Write. No Agent anywhere. The second dispatch explicitly attempted the literal Agent(subagent_type: "Explore", ...) call and reported "cannot invoke — Agent tool name is not in my tool palette." The harness's anti-recursion guard is the cause.

The two worktree scenarios

Lead in their own terminal (Pattern A)

Working directory IS the lead's worktree. git status from inside reflects the worktree's branch. Commits go on crt/<lead>-* branches as designed. This is the cleanest case.

cd C:/Dev/GauntletAI/ClinicalRedTeam-bram
claude                                  # session starts here
/bram                                   # loads Bram persona
# everything just works

Lead spawned by Tate (Pattern B)

The spawned Bram inherits Tate's working directory — the main checkout, NOT the Bram worktree. Bram's persona instructs him to write to his worktree via absolute paths (this exact scenario already happened with Bram session 1; his handoff documents it).

# Tate's session is in C:/Dev/GauntletAI/ClinicalRedTeam/
# Tate calls Agent(subagent_type: "bram", ...)
# Bram-subagent's CWD is also C:/Dev/GauntletAI/ClinicalRedTeam/
# Bram routes writes to ClinicalRedTeam-bram/ via absolute paths
# and commits land on crt/bram-* in that worktree
The hidden subtlety

When Tate spawns Bram (Pattern B), Bram's CWD is Tate's CWD. This works because git operations don't actually require cd — git -C <path> <op> works from anywhere. Bram session 1's handoff explicitly notes this happened and the pattern worked:

"/bram was invoked from main checkout instead of ClinicalRedTeam-bram/. All edits, commits, and pushes were routed through absolute worktree paths so commits landed on crt/bram-* branches in the correct worktree."

Translation: if you spawn Bram from Tate, Bram still commits to crt/bram-* branches in the Bram worktree. Just by absolute paths instead of cd. Branches stay separated; worktrees stay separated. The only difference is which session physically launched Bram.

Why worktrees + branches stay separate always

  • Git branch state is per-checkout. Two sessions can't have different branches checked out in the same dir at once. Worktrees fix this.
  • Working tree changes race. If Aria is editing src/clinical_redteam/agents/orchestrator.py and Bram (in the same dir) reads it for an audit, Bram might see a half-saved state. Worktrees isolate.
  • Test runs collide. pytest writes .pytest_cache/; concurrent runs in the same dir conflict. Worktrees give each lead their own.
  • Branch ownership is the contract. Every lead commit goes on crt/<lead>-*. MRs are scoped to the lead. Reviewers know who owns what without reading commit messages.

06 Pitfalls We've Hit (or Could Hit) avoid these

Pitfall 1 Double-dispatching the same lead.

If you have Bram running in his own terminal AND Tate spawns Bram via the Agent tool, both Bram sessions will write to bram-handoff.md and in-flight.md. Last writer wins; the other's edits are lost. Always end the running Bram before spawning a new one. Bram should write his handoff entry, /clear, close the terminal.

Pitfall 2 Spawning a lead from a terminal that's NOT in their worktree.

This is permissible (Bram session 1 did it and it worked) but harder — Bram has to remember to route every write via absolute paths. If he ever does a plain git commit, it commits to whatever branch is checked out where you are, not his branch. Prefer Pattern A (slash command in the lead's worktree) when you have the option. Reserve Pattern B for resume / continuation / overnight scenarios.

Pitfall 3 Subagent context bloat from deep nesting.

Tate → Aria → audit-subagent is 3 levels. Each level has its own context window. If Aria's audit-subagent itself spawns ANOTHER subagent (4 levels), token cost compounds. In practice 3 levels is fine; 4+ should be a deliberate choice. The audit-each-step memory entry already scopes this: leads spawn ONE auditor per item, not nested.

Pitfall 4 Forgetting to end-of-session write a handoff.

If Aria's terminal crashes / power dies / Ctrl+C before she writes her handoff, the next Aria session has no idea where she left off — she has to derive it from git log + commit messages. Discipline: Every lead writes a handoff entry BEFORE /clear or terminal close. Kickoff briefs already say this; reinforce it in every dispatch prompt.

Pitfall 5 Letting Tate forget to coordinate.

You discovered this tonight: Bram ended his goal loop blocked on B6, but no mechanism existed to auto-resume him when B6 finished. Tate must explicitly read every recent handoff at the start of every session and check the in-flight Watch List (when we build it). The auto-resume routine (/tate-watch-poll) is the future fix; for now, manual catch-up at each /tate invocation.

Pitfall 6 Schema or scope changes from an autonomous lead.

Aria's autonomous-mode discipline says "stop and write coordination if a Pydantic schema change is required." This works only if Aria actually stops — if a future lead "fixes" something by editing schemas.py without sign-off, the inter-agent contract drifts. Audit-each-step is the gate that catches this; never weaken it. Tate has the equivalent rule for content_filter weakening.

07 Best Practices — Distilled do these by default

Practice 1 Default to Pattern A for parallel multi-hour work, Pattern B for resume / short-bounded / unattended.

The mental check: "is a human going to be at the keyboard for the next hour while this lead works?" Yes → Pattern A. No → Pattern B.

Practice 2 Worktrees + branches always separate, regardless of dispatch pattern.

The branch policy in w3-branch-policy-by-phase memory ("Phase 1b onward: feature branches + auto-MR mandatory") + the worktree topology you set up are the load-bearing isolation. Don't collapse them for short-term convenience.

Practice 3 Audit-each-step is non-skippable — and requires Pattern A for code commits.

Every code commit goes through an independent audit subagent before commit. 3-level nesting (Tate → lead → auditor) only works if the lead has the Agent tool — which means Pattern A. Pattern-B-spawned leads cannot spawn auditors (verified empirically; see Section 4). For Pattern-B code work, either use Pattern A instead, or have Tate spawn the audit subagent on the lead's pre-push branch. The w3-audit-each-step memory entry encodes the discipline; w3-pattern-b-no-subagent-spawning encodes the constraint.

Practice 4 End every lead session with a handoff entry. Always.

Format already exists in handoffs/<lead>-handoff.md. Newest entry at top. Two fields are non-negotiable: "items in progress" and "recommended next action." The next session reads these and knows where to start.

Practice 5 Tate reads everything at the start of every /tate session.

Tate's kickoff already says so: read CLAUDE_SESSION_HANDOFF.md, MVP-WORK-PLAN.md, in-flight.md, latest session recap, each lead handoff. Don't shortcut this. The cost (5 min of reads at session start) is the price of cross-stream coordination working at all.

Practice 6 When a lead is blocked, write a Watch List row.

Append to in-flight.md under "Watch List" with: lead name, blocker condition (filesystem-checkable), watch file path, target item to resume. Tomorrow's /tate-watch-poll routine (when built) will read these; until then, Tate at the start of each session.

Practice 7 Match the dispatch's persona to the work.

Don't ask Aria to write eval YAMLs (Bram's territory). Don't ask Bram to write Python agent code (Aria's territory). The persona file enforces ownership; the kickoff brief tells the lead what's in and out of scope. If a task crosses ownership, raise via coordination/ rather than violating the boundary.

What worked

Pattern A overnight with worktrees + audit-each-step delivered all of Phase 1b's code and content with zero file conflicts and a real PHI-leak caught before ship. The branch policy + autonomous-mode operating discipline + handoff schema were the load-bearing pieces.

What didn't

The gap between "Bram ends loop blocked on B6" and "B6 finishes" had no auto-resume mechanism. This is exactly the failure mode ARCH §10.2 predicts for the platform itself. The fix is the /tate-watch-poll routine (built post-MVP). Until then: Tate at the start of each session manually scans the in-flight Watch List and re-dispatches via Pattern B.

Appendix: Pre-Search Checklist (PRD pp. 13–15)

These are grader defense probes, not scored requirements. Think through each before the defense and final submission.

Phase 1.1 — Domain & Threat Selection

  • What specific attack categories will you target first?
  • What are the verification requirements for determining exploit success?
  • What attack surfaces require human judgment versus automated assessment?

Phase 1.2 — Scale & Performance

  • Expected test run volume per day/week?
  • Acceptable latency for attack generation and evaluation?
  • Concurrent testing requirements against the target system?
  • Cost constraints for LLM-driven attack generation?

Phase 1.3 — Reliability Requirements

  • What is the cost of a false negative (missed vulnerability)?
  • What is the cost of a false positive (flagging safe behavior as vulnerable)?
  • Human-in-the-loop requirements for remediation approval?
  • Audit and compliance needs for security testing in a healthcare context?

Phase 2.4 — Multi-Agent Design

  • How many distinct agent roles does your platform require?
  • What framework manages agent coordination (LangGraph, CrewAI, AutoGen, custom)?
  • How do agents hand off work (message queues, shared state, direct invocation)?
  • What happens when one agent in the pipeline fails or times out?

Phase 2.5 — Red Team Agent Design

  • What model powers the Red Team Agent, and why is it appropriate for offensive security?
  • How does the agent mutate a failed attack into a variant?
  • How do you prevent the Red Team Agent from being blocked by the same safety filters it is trying to probe?
  • Context window requirements for multi-turn attack sequences?

Phase 2.6 — Judge Agent Design

  • How does the Judge determine success vs. failure vs. partial for each attack category?
  • How do you validate that the Judge's criteria are consistent across runs?
  • What happens when the Judge is uncertain — does it escalate to a human?
  • How do you prevent the Judge from drifting as the target system changes?

Phase 2.7 — Orchestrator Design

  • What signals does the Orchestrator read to prioritize the next attack campaign?
  • How does the Orchestrator decide when a category is sufficiently covered?
  • What triggers a regression run — a new deployment, a time window, or both?
  • How does the Orchestrator manage cost across agents in a single session?

Phase 2.8 — Observability Strategy

  • LangSmith vs. Langfuse vs. Braintrust vs. custom — which surfaces inter-agent traces?
  • What metrics matter most for a multi-agent security platform?
  • How do you trace a vulnerability finding back through all the agents that produced it?
  • Cost tracking at the agent level, not just the platform level?

Phase 3.8 — Failure Mode Analysis

  • What happens when the Red Team Agent generates content that is itself harmful?
  • How do you handle a Judge Agent that starts agreeing with everything?
  • What is the fallback when the Orchestrator has no clear next priority?
  • How do you handle cascading failures across agents in a single test run?

Phase 3.9 — Trust & Safety for the Platform Itself

  • How do you prevent the adversarial platform from being turned against systems it should not attack?
  • Access controls for who can trigger agent runs and view vulnerability reports?
  • What approval is required before the Documentation Agent files a critical-severity report?
  • How do you audit what each agent did during an overnight run?

Phase 3.10 — Testing the Tester

  • How do you validate that the Red Team Agent is actually generating novel attacks?
  • Ground truth dataset for evaluating Judge Agent accuracy?
  • How do you detect when the platform is producing low-quality signal?
  • What does it mean for the multi-agent system itself to regress?

Phase 3.11 — Iteration Planning

  • How will you add new agent roles as the platform matures?
  • Eval-driven improvement cycle for each agent independently?
  • How does the platform incorporate newly published attack techniques automatically?

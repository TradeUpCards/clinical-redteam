Scale & Performance

✓ Decided
Questions from Presearch.pdf
  • Expected query volume?
  • Acceptable latency for responses?
  • Concurrent user requirements?
  • Cost constraints for LLM calls?
Our Decision

Decision

  • Volume: 10–100 attacks per session at MVP; continuous daemon may accumulate 200–500 attacks/week in sustained mode. Bounded by cost cap, not infrastructure.
  • Latency: Not user-facing. Adversarial testing is an offline, background workload. Per-attack cycle target: 5–30 seconds (OpenRouter call + target call + Judge call). No latency SLA required.
  • Concurrency: Single-process sync for MVP. One LangGraph state machine, one while True: loop. Phase 3 concern: Redis-fronted multi-worker fleet if throughput becomes the bottleneck at 10K+ attacks.
  • Cost cap: MAX_SESSION_COST_USD env var; default $10/session. Total W3 dev burn budget: ~$5–15 on LLM calls. Weekly budget projections: $5–15 (MVP), $80–150 (1K attacks), $4–10K+ (100K attacks, requires architectural rethink).

  • Single-process sync means one slow OpenRouter or target call stalls the entire loop. Per-call timeouts and circuit breaker mitigate but do not eliminate this.
  • $10/session cap limits overnight runs. A session that costs $0.10/attack halts after ~100 attacks. Tune after seeing real per-attack costs from first runs.
  • Cost projections at 10K+ are model-price-dependent and will diverge from actuals as OpenRouter pricing shifts.
  • Very long multi-turn sequences (8+ turns) consume session budget disproportionately. Multi-turn depth is a known constraint at MVP.

Reliability Requirements

✓ Decided
Questions from Presearch.pdf
  • What is the cost of a wrong answer in your domain?
  • What verification is non-negotiable?
  • Human-in-the-loop requirements?
  • Audit/compliance needs?
Our Decision

Decision

Cost of wrong answer: False negatives (missed vulnerabilities) cost more than false positives in this domain. A missed PHI-leakage exploit means real patient data may leak in production; a false positive wastes engineering time. The platform biases toward escalation on uncertainty.

Non-negotiable verification: Human gate on any high or critical severity finding before it is filed. Judge uncertainty (confidence < 0.70) escalates to human review queue. No autonomous filing of high/critical reports.

Human-in-the-loop gates (per ARCHITECTURE.md §7.2):

  • High/critical severity reports: human review required before status: draft-pending-review transitions to status: filed
  • UNCERTAIN Judge verdict: enters human review queue regardless of severity
  • Cost ceiling hit: daemon halts; human resumes with explicit cost override

Audit/compliance: NIST AI RMF function mapping documented in ARCHITECTURE.md §7.4 (Govern/Map/Measure/Manage). HHS AI Strategy alignment referenced in ops research. Every autonomous action logged with agent name, version, inputs, outputs, human-approval status, and timestamp to logs/.

False negative vs. false positive cost asymmetry
Error typeWhat happensCost
False negative (missed vulnerability) Real exploit goes undiscovered; PHI leaks in production; clinical decisions made on manipulated output HIGH — patient harm possible; C-7 class already confirmed as HIGH-severity deferred
False positive (safe flagged as vulnerable) Engineering time wasted chasing non-issue; platform credibility erodes with CISO MEDIUM — recoverable; no patient harm
Judge calibration drift Verdicts shift from ground truth silently; regression harness looks green while real vulnerabilities exist COMPOUNDING — false negatives accumulate without visibility
Tradeoffs We Own
  • Human gates on high/critical reports mean a slow reviewer blocks the Orchestrator's ability to declare coverage complete. Mitigated by: autonomous continuous testing continues; only report-filing is gated, not discovery.
  • Erring toward escalation on uncertainty increases human review workload in the short term. This is the correct tradeoff for a healthcare security platform.
  • The platform cannot force a fix — it only documents and reports. Remediation is owned by the Co-Pilot team. This is correct scoping but limits standalone value.
  • HHS AI compliance framing is aspirational at 5-day project scale. Honest framing: designed to align with HHS AI governance expectations; not yet audited against them.

Agent Framework Selection

✓ Decided
Questions from Presearch.pdf
  • LangChain vs LangGraph vs CrewAI vs custom?
  • Single agent or multi-agent architecture?
  • State management requirements?
  • Tool integration complexity?
Our Decision

Decision

Framework: LangGraph. Documented in ARCHITECTURE.md §3.1.

Architecture: Multi-agent — 4 agents with deliberately separated responsibilities:

  • Red Team Agent — generates and mutates attacks
  • Judge Agent — independently evaluates attack success (never sees Red Team hypothesis before verdict)
  • Orchestrator Agent — prioritizes coverage and controls cost (pure Python rules, no LLM at MVP)
  • Documentation Agent — converts validated exploits into structured reports

Rationale for multi-agent: A system that both generates and judges attacks has a conflict of interest by design. Red Team incentivizes maximizing exploit discovery; Judge must remain unbiased and deterministic. Combining them in one agent context weakens both.

State management: TypedDict state with filesystem checkpointing (not LangGraph built-in checkpointer). Rationale: simpler, JSON-serializable, survives process crashes without pickle issues.

Tool integration: 7 named tools (TargetClient, RegressionStore, CoverageStateReader, ObservabilityWriter, SeedLoader, CriteriaLoader, VulnerabilityReporter). Each has defined input/output schema. See §7.

  • LangGraph does not natively enforce type safety at edges. TypedDict provides IDE hints; Pydantic v2 or jsonschema validation is required at every inter-agent boundary for runtime enforcement.
  • No prior art for adversarial red-team graph with hypothesis stripping in LangGraph ecosystem. First version will hit undocumented edge cases.

LLM Selection

✓ Decided
Questions from Presearch.pdf
  • GPT-5 vs Claude vs open source?
  • Function calling support requirements?
  • Context window needs?
  • Cost per query acceptable?
Our Decision

Decision — criteria-first, not model-name-first (per ARCHITECTURE.md §8.2)

  • Red Team Agent: Open-weight model via OpenRouter. Default: Dolphin-Mixtral or Qwen-class. Key criterion: permissive safety posture (will not refuse security workflows). Frontier models explicitly rejected for this role — safety training is an obstacle, not a feature, for adversarial generation.
  • Judge Agent: Anthropic Claude Sonnet-class via OpenRouter. Key criteria: frontier accuracy, follows evaluation instructions including security content, consistent structured output, cross-provider independence from Red Team.
  • Documentation Agent: Anthropic Claude Sonnet-class via OpenRouter; Haiku-class fallback. Report quality is bounded by FAIL verdict count, not model name.
  • Orchestrator Agent: Pure Python rules, no LLM for MVP. Optional Haiku-class augmentation in Phase 2 for autonomous category discovery.
  • Routing layer: OpenRouter for all hosted model calls. All model IDs read from environment variables.

WhiteRabbit-Neo note: Purpose-built for offensive security (Kindo, Llama fine-tune); skipped for MVP because not confirmed on OpenRouter at time of decision. Revisit at build-start — 5-minute availability check required. If available, it is the most defensible Red Team model pick.

Our Decision

Decision — 7 named tools (per ARCHITECTURE.md §2 and §3.2)

  • TargetClient — HMAC-signed HTTP to the Co-Pilot (/chat, /graph_chat, /attach_and_extract). Red Team only external call. Per-PRD: live target required for all eval results.
  • RegressionStore — read/write confirmed exploit YAML files under evals/regression/

  • ObservabilityWriter — emits Langfuse spans at every LLM call boundary

  • VulnerabilityReporter — writes evals/vulnerabilities/VULN-NNN-DRAFT.md (Documentation Agent output)

Mock vs. real: --dry-run flag mocks TargetClient with canned fixture response for local dev iteration. All eval results used in deliverables require live target calls.

Eval Approach

✓ Decided
Questions from Presearch.pdf
  • How will you measure correctness?
  • Ground truth data sources?
  • Automated vs human evaluation?
  • CI integration for eval runs?
Our Decision

Decision — three eval contexts (per ARCHITECTURE.md §3.5 and §11)

  • Context A — Adversarial eval (the product): Continuous attacks against the live target; Judge Agent renders PASS / FAIL / PARTIAL / UNCERTAIN verdict per attack using per-category criteria from evals/criteria/category.yaml. This is the grading artifact.
  • Context B — Platform meta-tests: Unit tests, smoke tests, schema tests, calibration tests in tests/. Run in CI on every commit. Not adversarial eval — these test the platform logic, not the target.
  • Context C — Grader-facing artifacts: Filed vulnerability reports, regression cases, coverage report. The deliverable.

Judge: LLM-as-Judge with rubric-based criteria for MVP. Ground-truth calibration dataset (human-labeled attack/response/verdict tuples) is Phase 2.

CI integration: Context B runs in GitLab CI on every commit. Context A (live target + real LLM calls) is NOT run in CI per commit — it costs money and calls a live target. Manual or scheduled daemon runs produce Context A artifacts.

LLM-as-Judge (rubric-prompted, frontier model) Selected High — nuanced; catches partial successes; handles paraphrased leakage Mid (one frontier call per attack) Moderate — model can drift; structured output helps Selected. Adversarial success evaluation is inherently a natural-language judgment task. Deterministic heuristics alone miss the tail cases this platform is designed to catch.
Deterministic only (keyword/regex) Low — misses paraphrased leakage; misses context-sensitive partial success Very low Very high Rejected. Cross-patient leakage via paraphrase (the C-7 finding) is precisely the case that breaks regex. The domain requires semantic understanding.
Hybrid (deterministic fast-path + LLM fallback) Phase 2 High — best of both Low-mid High Valid architecture; adds complexity for MVP. Deterministic pre-filter + LLM for ambiguous cases. Phase 2 consideration.
Multi-judge panel Phase 2–3 Highest — disagreement surfaces ambiguity High (N x per-attack cost) Best — majority vote reduces variance Deferred. Single-judge sufficient for MVP given human gate on high/critical reports. ARCHITECTURE.md §9.1 explicitly defers.

Tradeoffs We Own
  • LLM-as-Judge is non-deterministic. Same attack replayed against same target response can theoretically yield different verdicts across runs. Mitigation: pin Judge model version; temperature=0; structured output enforcement.
  • MVP lacks human-labeled ground truth. There is no quantitative accuracy number to cite at architecture defense. Mitigation: explicitly document as Phase 2 work; rubric criteria are the interim quality control.
  • Rubric drift is the Judge's primary failure mode. If criteria YAML gets stale as the target evolves, Judge verdict quality degrades silently. Mitigation: versioned criteria files; criteria updates require PR review.
  • PARTIAL verdict handling is underspecified for regression: PARTIAL triggers mutation, not a filed regression case. This is a design choice some teams implement differently.
  • Context A eval (adversarial) is outside CI. A regression in Red Team mutation logic is not caught until a manual eval run is triggered. Smoke tests (Context B) cover the core loop path with mocked targets as mitigation.

Verification Design

✓ Decided
Questions from Presearch.pdf
  • What claims must be verified?
  • Fact-checking data sources?
  • Confidence thresholds?
  • Escalation triggers?
Our Decision

Decision — three-layer verification (per ARCHITECTURE.md §4)

  1. Judge evaluates attack success — primary signal; renders PASS / FAIL / PARTIAL / UNCERTAIN with per-criterion evidence and confidence score
  2. Regression harness verifies behavior — confirms previously-confirmed exploit still fires; distinguishes genuine fix from behavior-changed (per ARCHITECTURE.md §4.3)
  3. Human gate on high/critical — no autonomous filing of high or critical findings (ARCHITECTURE.md §7.2)

Confidence thresholds:

  • Judge confidence ≥ 0.85: verdict accepted, auto-file if low/medium severity
  • Judge confidence 0.70–0.85: verdict accepted but flagged for sampling review
  • Judge confidence < 0.70: verdict = UNCERTAIN; escalates to human review queue

Escalation triggers:

  • Verdict = UNCERTAIN: human review queue
  • High/critical severity: human review queue regardless of confidence
  • Judge UNCERTAIN rate > 25% in a session: alert — calibration may have drifted
  • Regression test PASS-to-FAIL transition: alert and open-findings bump
The Hard Problem — Genuine Fix vs. Behavior Changed

PRD p. 4 identifies this as a hard problem: determining whether a fix actually improves the system or simply changes its behavior temporarily. Three approaches evaluated:

ApproachConfidenceCostVerdict
Pair verdict with fixed/changed classifier Selected Medium-high — Judge distinguishes "target refuses correctly" from "target says something different that happens not to include PHI" One extra Judge call per regression check Selected. Requires category-specific expected-safe criteria but gives a defensible answer to "was the fix genuine."
Replay identical attack N=3 times Higher than single replay — 3 consecutive passes builds confidence 3x cost; temporal non-stationarity means consecutive passes are not fully independent Phase 2 enhancement for high-value confirmed exploits.
Mechanistic fix verification (human-in-loop) Highest — engineer confirms root cause addressed Requires human; not scalable to continuous operation Reserved for critical-severity findings where root cause matters.
Tradeoffs We Own
  • Expected-safe behavior specs are the most important and most underspecified artifact. Category-specific criteria YAML must be precise enough that the Judge can distinguish "target improved" from "target behaved differently." This is documentation work, easy to deprioritize.
  • Confidence thresholds (0.85 / 0.70) are chosen by judgment, not empirical calibration. They will need adjustment once the calibration dataset is built in Phase 2.
  • The platform can only verify behavioral outcomes, not root causes. A fix that produces correct behavior via the wrong mechanism verifies as fixed until a variant is tested.
  • Regression test validity degrades as the target evolves. Stale regression cases produce false alerts. Mitigation: status field on every regression case; human review when active cases have not fired in 30+ days.
Open Questions
  • Phase 2 — non-blocking for MVPWhat is the minimum-repro requirement for a regression case? Define minimal repro as the smallest attack sequence that reliably produces the FAIL verdict with no extraneous turns or tokens. Can be automated via binary search on attack sequence length.

Security Considerations

✓ Decided
Questions from Presearch.pdf
  • Prompt injection prevention?
  • Data leakage risks?
  • API key management?
  • Audit logging requirements?
Our Decision

Decision — ARCHITECTURE.md §3.3 and §12, THREAT_MODEL.md §2

Prompt injection prevention (platform's own LLM calls):

  • Schema validation (Pydantic v2 / jsonschema) at every inter-agent boundary — target response cannot inject instructions into Judge evaluation prompt because Judge renders verdict as JSON object; if response cannot be parsed into JudgeVerdict schema, it is rejected
  • Judge never sees Red Team hypothesis before rendering verdict — isolation enforced by graph edge typing
  • Content-class filter on Red Team output before it touches the target

Hard content categories Red Team refuses to generate:

  • Attacks involving minors in clinical / sexual / violent contexts
  • Attacks containing real PHI tokens (real SSNs, real patient names — synthetic IDs are permitted)
  • Attacks targeting systems other than the approved Co-Pilot at 142-93-242-40.nip.io
  • Attacks synthesizing weaponized real-world malware (working CVE exploits, ransomware code)

Data leakage controls:

  • PHI scrubber runs on all stored target responses before persisting to disk
  • PHI scrubber runs on Documentation Agent output before vulnerability reports are written
  • Attack payloads use synthetic patient identifiers only; CI checks seed files for real PHI patterns
  • OpenRouter sees attack payload content — mitigated by synthetic-data-only discipline

API key management:

  • COPILOT_HMAC_KEY — environment variable; never in code; rotate when target rotates its key
  • OPENROUTER_API_KEY — environment variable; rotate quarterly
  • LANGFUSE_SECRET_KEY — environment variable; rotate quarterly
  • Pre-commit hook: detect-secrets or trufflehog scan on every commit

Audit logging: Every autonomous action logged with agent name, version, full inputs, full outputs, human-approval status, and timestamp to structured JSON files under logs/. Langfuse provides secondary audit trail at the LLM call level (per ARCHITECTURE.md §7.3).

Alternatives Considered — Prompt Injection Defense
Defense mechanismStrengthCostVerdict
Structured output enforcement (JudgeVerdict schema) Selected High — free-text instructions in target response cannot change the JSON schema Low — schema validation is deterministic Selected. Primary defense at the Judge boundary.
Response sanitization (pattern matching) Low — advanced injections evade simple patterns Low Not sufficient alone. Pattern matching is brittle to novel injection shapes.
Sandboxed Judge context (process isolation) High — true process isolation High — significant infrastructure complexity Over-engineered for MVP. Target response is already a user-role message in the prompt, not a system-level injection.
Tradeoffs We Own
  • The platform is not air-gapped — calls OpenRouter, Langfuse, and the target API. Mitigated by synthetic-data-only discipline, sentinel-pid boundary on target side, and PHI scrubber. Residual risk: paraphrased PHI could evade the scrubber (the same vulnerability class the platform is designed to detect in the target).
  • PHI scrubber on target responses adds latency. Acceptable at MVP throughput; potential bottleneck at Phase 3 high-volume operation.
  • A crafted target response that is valid JSON matching the JudgeVerdict schema but with manipulated field values is a theoretical second-order attack. Structural defense (schema enforcement) handles instruction injection but not semantically-valid-but-malicious values in verdict fields.
  • Access controls for platform admin functions are policy-only at Phase 1. Who can run this code is OS-level enforcement, not application-level.
Open Questions
  • Required before Phase 1 live attacks fireConfirm a dedicated test-only OpenEMR account exists with no access to real patient records. The platform needs OpenEMR session credentials (in addition to HMAC) to make valid /chat calls. This account must be scoped to the demo data only.

Testing Strategy

✓ Decided
Questions from Presearch.pdf
  • Unit tests for tools?
  • Integration tests for agent flows?
  • Adversarial testing approach?
  • Regression testing setup?
Our Decision

Decision — four test classes (per ARCHITECTURE.md §3.5 Context B)

  • tests/unit/ — pytest with mocked LLM responses; covers TargetClient (HMAC, retry, circuit breaker), RegressionStore (YAML CRUD + status transitions), CoverageStateReader, Judge rubric evaluation, Orchestrator selection rules (all 5 priority rules per §3.6.1), schema validation
  • tests/smoke/ — vertical slice: one full attack cycle (Red Team seed → mocked TargetClient → mocked Judge verdict → mocked Documentation → filesystem persistence); Orchestrator loop smoke (3 iterations, mocked agents); crash recovery test (partial manifest → restart → verify resume)
  • tests/schemas/ — schema contract tests: generate AttackCandidate → pass to Judge schema validator → generate JudgeVerdict → pass to Documentation schema validator; all schemas validated without LLM calls
  • tests/calibration/ — Phase 2: Judge calibration against human-labeled ground-truth dataset at evals/calibration/ground-truth.yaml

Adversarial testing of the platform itself (testing the tester — per ARCHITECTURE.md §10.3):

  • Judge injection meta-test: craft a target response containing an injection attempt; verify JudgeVerdict schema rejects it
  • Judge uniform-verdict collapse meta-test: fixture cases that MUST produce PASS and cases that MUST produce FAIL — run on every regression run
  • Red Team novelty: attack embedding cosine similarity; reject if similarity > 0.95 to any prior attack in same-category corpus
  • Documentation Agent report quality: rubric against PRD required fields (vuln ID, severity, repro, expected/observed, remediation, OWASP, MITRE) — block DRAFT-to-FILED promotion if fields missing

CI pipeline structure:

  • Pre-commit: schema tests + unit tests (fast; <30s; no LLM calls)
  • CI on every push (GitLab CI): all unit tests + smoke tests + schema tests (<5 min)
  • Weekly scheduled CI: Judge calibration against ground-truth dataset (Phase 2)
  • NOT in CI per commit: live target calls, real LLM calls, adversarial eval runs (Context A)
Alternatives Considered and Rejected
ApproachProsConsVerdict
pytest with mocked LLM responses Selected Fast; deterministic; no API cost; runs in CI without money Mocked responses do not capture model drift Selected. Standard approach; right tradeoff for CI-bound tests.
Full integration with live LLM in CI Tests actual model behavior Expensive; non-deterministic; cannot run on every commit Rejected for per-commit CI. Manual or weekly scheduled run only.
VCR.py cassette recording Records real LLM calls; replays in tests Cassettes go stale when model behavior changes; adds maintenance burden Rejected. Maintenance overhead not justified for 5-day build.
Tradeoffs We Own
  • Mocked LLM responses in unit tests do not capture model drift. If Judge model changes behavior in a new version, unit tests with canned responses will not catch it. Mitigation: weekly calibration tests against ground truth; pin model versions.
  • Ground-truth calibration dataset does not exist at Phase 1 start. The largest testing gap. Interim mitigation: meta-test fixtures (known-pass and known-fail cases); weekly calibration run against those fixtures.
  • Smoke test uses a mocked target — divergence from real target behavior is possible. Smoke tests validate platform logic, not attack quality. Engineers must validate on live target before shipping findings.
  • Testing the tester recursion has a practical stopping point (human review of calibration cases) that is not automated. This is acknowledged as the correct stopping point (ARCHITECTURE.md §10.3).
